What Gets Built — Detail
1. URL-Synced Filters (the Linear pattern)

All filter state lives in the URL (?status=PENDING,MISSED&platform=ZOOM&from=...), read server-side in page.tsx and passed as initial filters to the client list — this means filters are shareable links and survive refresh, and the RSC can do an initial server-rendered fetch (no loading flash on first paint).
useMeetingFilters is a thin wrapper around useSearchParams/useRouter (or nuqs if added) that exposes typed getters/setters — every filter control reads/writes through this hook, never local component state alone.

2. MeetingFilters bar

A horizontal row directly under PageHeader: a Popover-based multi-select for Status (checkbox list inside the popover — using shadcn Checkbox rows), a single-select DropdownMenu for Platform, a date-range trigger (simple two-input popover for now, full DatePicker polish later), and a free-text search input (debounced via existing useDebounce).
Active filters render as removable FilterPill chips below/beside the bar — click × removes that filter and updates the URL.

3. Dense List, Not Cards

Deliberate departure from "card grid" — MeetingListHeader + MeetingListRow form a table-like dense list (36px rows), columns: Status dot, Title (truncate + platform icon inline), Platform badge, Scheduled time (RelativeTime), participant count, chevron.
Every row: cursor-pointer, hover bg, entire row is a <Link> — clicking anywhere navigates to /meetings/:id. This is the canonical "dense info, every item clickable" pattern that Commitments and Action Items will copy on Days 33/36.
Keyboard: list supports ↑/↓ to move a focus ring between rows and Enter to open (small custom hook, reused later) — this is what makes it feel like Linear instead of a generic web app.

4. Cursor Pagination

useMeetings uses TanStack Query's pattern for cursor pages; UI shows a "Load 20 more" text button at the bottom (CursorPagination) rather than numbered pages — consistent with the backend's cursor-based contract from Day 17/19.

5. Empty State

MeetingEmptyState: no illustration, no AI-vibe copy — just a centered block: icon (outline, monochrome), "No meetings yet" (13px), one-line subtext, a single primary button "Add meeting" (opens the Day-32 stub AddMeetingModal as a Sheet).

6. Status & Platform visuals

MeetingStatusBadge: a single shared color map (SCHEDULED = neutral, BOT_JOINING/RECORDING = amber-ish pulse dot, DONE = neutral-green text, FAILED = red text, CANCELLED = muted strikethrough-style) — all low-saturation, text-forward badges (shadcn Badge variant outline), never solid color blocks.
