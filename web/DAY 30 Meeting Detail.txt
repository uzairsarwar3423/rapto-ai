What Gets Built — Detail
1. Virtualization (the core engineering challenge)

VirtualList.tsx is built as a generic, reusable virtualizer wrapper (estimateSize, renderItem, overscan props) — not transcript-specific — because Day 33+ Commitments/Action Items lists at scale will eventually want the same primitive. This is the platform-wide investment of the day.
TranscriptViewer consumes VirtualList with estimateSize={() => 72} per turn, overscan={8}. Only visible turns + overscan are mounted in the DOM regardless of transcript length — this is the only way a 3-hour, 1000-turn all-hands transcript stays scrollable at 60fps.
Container is wrapped in a ScrollArea for consistent scrollbar styling matching the rest of the app (not the raw browser scrollbar).

2. TranscriptTurn (dense row)

Avatar (small, 24px) + speaker name (bold 13px) + relative timestamp (mm:ss from meeting start) + the spoken text below, left-aligned, no chat-bubble background — reads like a transcript document, not a messaging app. Unresolved speakers (confidence < 0.8, per the backend design) render with a muted "External participant" label instead of guessing a name.

3. TranscriptSearch (sticky, in-transcript)

A sticky bar pinned to the top of the transcript panel (below the tab bar): text input + live match count ("3 of 47") + prev/next chevron buttons (keyboard: Enter/Shift+Enter cycle matches) — calls the backend's Atlas Search-backed query (per existing API design) when the query is non-trivial, but does instant client-side substring matching for typed-character feedback before debounce fires, so it never feels laggy.
On finding a match, the virtualizer scrolls to that index (virtualizer.scrollToIndex) and TranscriptSearchHighlight wraps the matched substring in a subtle highlight span (background tint, no bright yellow marker-style highlight — stays in the neutral palette).

4. TranscriptSpeakerFilter

A DropdownMenu listing all participants; selecting one filters the virtualized list to only that speaker's turns (client-side filter over the already-fetched transcript, since transcripts are fetched whole, not paginated, per the existing data design).

5. Empty/Guard States

If meeting.status !== 'DONE' or mongoTranscriptId is null, render TranscriptEmptyState ("Transcript will appear once processing completes") instead of attempting a fetch — matches the backend's documented TRANSCRIPT_NOT_AVAILABLE error code, handled gracefully rather than as a console error.
